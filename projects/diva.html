$ cat rule_design.md

Hard rules (supplier fraud, duplicate payment) cause immediate rejection without evaluating remaining rules. Soft rules (amount tolerance, date grace period) are all evaluated and bundled into a single structured exception report.

Finance team adjusts thresholds via YAML config without code deployment — critical for seasonal needs (e.g., higher tolerance during Q4 budget close).

$ cat llm_extraction.md

Non-standard invoices (Chinese suppliers, German language, handwritten) use an LLM extraction pipeline as fallback when structured ERP/EDI data is unavailable:

  • pymupdf → native page text extraction
  • pymupdf4llm → markdown layout with table structure
  • pytesseract → OCR with multilingual support (chi_tra+chi_sim+deu+eng)
  • LLM Call 1 → extract line-item table as CSV (Row No., SP No., PCS, Amount, Unit Price)
  • LLM Call 2 → extract header fields as JSON (bill_to, currency, invoice_number, invoice_date)
  • LLM Call 3 → JSON correction pass for malformed output

$ cat why_rule_based.md

  • Auditability: Finance auditors require explainable decisions — "rule X failed with value Y" vs. "model confidence 0.73"
  • Data volume: Limited labeled invoice-PO pairs at launch — insufficient for reliable ML classifier
  • Configurability: Finance can adjust tolerances without ML retraining
  • Early ML baseline underperformed the rule engine by 15 percentage points
  • Future ML ready: rule abstraction layer designed so ML could replace line-item matching later
