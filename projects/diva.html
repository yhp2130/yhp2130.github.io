Use Case — Finance Controller Distributor Reconciliation
stakeholder: finance controller · domain: distributor point-of-sale reconciliation

Distributors submit monthly invoices for semiconductor products they have sold to end customers. The Finance Controller needed to reconcile these invoices against the company's internal Sales & Marketing Data (SMD) from the ERP — verifying that quantities, prices, currency, and customer identity reported by the distributor matched the company's records.

The process was fully manual: controllers downloaded invoice PDFs, cross-referenced each line item against ERP exports, and flagged discrepancies by hand. With 100+ invoices per month from distributors across multiple countries and languages, this was a significant recurring burden.

The LLM pipeline handled multilingual invoices (Chinese, German, English) via pytesseract with chi_tra+chi_sim+deu+eng language packs and pymupdf4llm for layout-aware text extraction. The LLM was called three times per invoice: once for line-item table extraction (semicolon-delimited CSV), once for header field extraction (JSON), and once to correct malformed JSON output.

LLM vs ABBYY — Extraction Quality Comparison
decision: ABBYY chosen for production · LLM track discontinued

The on-prem LLM produced inconsistent extraction results — field hallucination, variable JSON structure, and poor handling of scanned invoices with low OCR confidence. Consistency degraded further for multi-page invoices and non-standard layouts.

ABBYY Vantage — using a trained document skill (combined_plank3) — produced structured, consistent field extraction across invoice formats. The pre-trained skill handled layout variation without requiring prompt engineering or retry logic.

Decision: ABBYY chosen as the production extraction engine. LLM approach retained as fallback reference but not deployed. Both codebases maintained in the repository.
Rule Engine — Matching Logic & ABBYY Integration
built: diva_rules.py · 6 matching flags · fuzzy + LLM company matching

Designed and implemented the rule matching engine (apply_rules()) that compares ABBYY-extracted invoice data against ERP sales data. Each invoice produces a structured flag set:

FlagLogic
smd_flagInvoice line item found / not found in ERP sales data
currency_flagInvoice currency matches ERP reference currency
quantity_flagPer-line quantity deviation check (invoice vs ERP)
price_flagPer-line price comparison — mean / max / min
sap_number_flag4-state: -1 not found · 0 no match · 1 exact · 2 found in list
date_flagInvoice date matches ERP expected date

Company name matching required a two-strategy approach — distributor names vary by abbreviation, legal suffix (Ltd/Co./GmbH), and romanization of CJK characters:

  • ASCII names: thefuzz.partial_ratio — ≥80 match · 30–79 partial (human review) · <30 mismatch
  • CJK names: LLM semantic comparison — fuzzy matching fails on Traditional/Simplified/mixed scripts
Design choice: Rule-based over ML — finance audit requires explainable decisions ("field X mismatched by Y%") not model confidence scores. Rules are also configurable without retraining.
Production Deployment
stack: ABBYY API + FastAPI + Elasticsearch · containerised on Docker

The production pipeline (main_abbyy.py) runs on a schedule: scans the invoice drop folder, calls the ABBYY Vantage API for each PDF, passes extracted data through apply_rules(), and posts structured match results to Elasticsearch via Logstash for the Finance Controller's review dashboard.

A Vue.js frontend surfaces exceptions — invoices with flag mismatches — for human review. Fully matched invoices are auto-approved and routed for payment processing. The system is containerised with Docker and served via FastAPI.

Outcome: Finance Controller reconciliation automated for standard invoices. Exception reports replace manual cross-referencing. Full audit trail stored in Elasticsearch for compliance.
