Logistics contracts are reviewed per cycle — each containing dozens of clauses covering liability caps, penalty terms, SLA obligations, and payment conditions. Manual review took days per cycle and required specialist knowledge. The goal: automatically classify and extract key clause content from scanned PDF contracts.

ICC Takeover — Regex + LDA Predecessor
predecessor: regex rules + Latent Dirichlet Allocation

Took over from a colleague. The existing system used regex pattern matching combined with Latent Dirichlet Allocation (LDA) for topic modelling on OCR-converted contract text.

Results were poor due to two compounding issues:

  • OCR noise from scanned copies — spotted pages, skewed text, poor contrast — degraded input quality
  • Legal clause complexity — clauses are long, nested, cross-referencing; LDA topic coherence broke down
Decision: Abandon LDA. Move to supervised transformer-based classification with clean text extraction.
Model Experimentation — BERT, RoBERTa ± CUAD
experiments: bert-base-uncased · roberta-large · roberta-large-mnli · CUAD domain pre-training

Explored encoder-only transformers for clause classification. Tested four configurations:

ModelTrainingObservation
BERT baseFine-tuned on labelled contractsBaseline — reasonable but lower F1
RoBERTa largeFine-tuned on labelled contractsBetter generalisation across clause styles
RoBERTa + CUADDomain pre-trained on Contract Understanding Atticus DatasetMarginal improvement — logistics contracts differ from CUAD's US legal style
RoBERTa-large-MNLIZero-shot + fine-tuned (natural language inference head)Best overall — selected for production
Insight: MNLI pre-training gave RoBERTa strong entailment reasoning useful for matching clause text to topic labels — even before fine-tuning.

Transcription — Nougat over OCR
model: Nougat (Meta) · replaces: EasyOCR / pytesseract

Contract PDFs are structurally similar to academic papers — dense paragraphs, numbered clauses, multi-column layouts, embedded tables. Standard OCR (EasyOCR, pytesseract) was sensitive to scanning noise: spots, skew, low contrast.

Switched to Nougat (Meta's neural optical understanding for academic documents), which treats the page as an image and generates clean text without a separate OCR step. Nougat's post-processors cleaned LaTeX artefacts, repeated lines, and headers/footers.

Pipeline pre-processors: Chinese character removal (for bilingual contracts), table extraction via DETR (tables overpainted white before Nougat to prevent misread), header/footer removal, repetition deduplication.
SetFit vs Supervised Fine-tuning — Complexity Trade-off
experiment: SetFit (few-shot) vs standard fine-tuning

Tested SetFit (sentence-transformer + few-shot contrastive fine-tuning using all-mpnet-base-v2) against standard supervised fine-tuning of RoBERTa. Both approaches produced comparable F1 on the multilabel multiclass classification task.

Decision: Chose standard supervised fine-tuning — simpler training pipeline, no contrastive pair generation, easier to maintain in production. SetFit offers an advantage mainly when labelled data is scarce; with 200 labelled contracts the benefit did not justify the added complexity.
Pipeline Design — Parallel Transcribe + Classify + Extract
stack: DETR → Nougat → RoBERTa → BART → Oracle DB

Pipeline processes contracts in page batches to reduce GPU idle time:

StageComponentRole
1. Table detectionDETR (detr-doc-table-detection)Detect & extract table images; overpaint white in document so Nougat reads text only
2. TranscriptionNougatConvert page images → clean paragraph text; post-process: dedup, LaTeX removal, line filtering
3. ClassificationRoBERTa-large-MNLI (fine-tuned)Multilabel multiclass: assign each sentence to one or more of 8 clause topics
4. ExtractionBART (seq2seq)Summarise classified clause text into structured key-value output per topic
5. StorageOracle DBPersist extracted clauses with contract ID, topic, sentence, and confidence score

CAA handling: amendment contracts overwrite prior BSA clauses — only amended topics are present, so missing topics fall back to BSA values stored from the previous cycle.

Production Deployment — Human-in-the-Loop
deployed to OpenShift

Results shared with business users. Deployed to production with a human-in-the-loop admin interface (Vue.js + Django REST): admins can review, amend, and approve AI-extracted clause data before it is used downstream.

Outcome: F1 = 0.8 weighted across 8 topics. Contract review cycle reduced from days to minutes.
Tool Conversion — ClaireGPT Integration
wrapped as LangChain tool for agentic use

BART was later wrapped as a LangChain tool within the ClaireGPT agent, enabling contract clause queries through natural language. Users could ask "What is the warranty period for supplier X?" and the agent would invoke the BART tool to retrieve and summarise the relevant clause.

💡 Retrospective Learning — GPT-4.1 Experiment

After deployment, ran a comparative experiment using GPT-4.1 for clause extraction on the same contract set. Results were comparable to the fine-tuned pipeline. Key takeaway: for low-volume structured extraction from varied document formats, multimodal LLMs are now the pragmatic first choice. The fine-tuned pipeline remains valuable for high-volume, on-prem deployments where API cost and data privacy are constraints.
