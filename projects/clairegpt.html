CODEC — Student FYP Takeover: Hybrid Semantic Search
stack: Django · SentenceBERT · TF-IDF · image corpus · OpenShift

Took over a student final year project implementing a hybrid semantic search engine for logistics knowledge documents. The system combined SentenceBERT semantic embeddings with TF-IDF keyword scoring:

combined_score = k_coeff x sbert_score + (1 - k_coeff) x tfidf_score

Documents extracted from PowerPoint (pptx) and Word (docx) using python-pptx / python-docx. Computer vision built a parallel image corpus — slide diagrams and images were separately embedded and searchable alongside text. A supervised click and rating boost applied score adjustments based on past user interactions with similar queries (lightweight collaborative filtering without retraining).

Limitation: All tensors loaded in-memory at startup (pickled embedder, corpus_embeddings, tfidf_matrix). Did not scale past ~1,000 docs. No LLM synthesis — returned raw page snippets.
CODEC 2.0 — Linux Refactor and GPU Acceleration
trigger: OpenShift has no Windows container support · added CUDA inference

OpenShift does not support Windows containers. The original Django app was built on Windows-specific paths and dependencies. Refactored the entire codebase to be Linux-compatible: path handling, file encoding, model loading, and service startup updated for container-native Linux environments.

Added GPU acceleration for SentenceBERT inference: torch.device("cuda:0" if torch.cuda.is_available() else "cpu"). Corpus embeddings and query encoding moved to GPU, reducing search latency for long queries. Core hybrid algorithm unchanged.

Outcome: Production-deployed on OpenShift. Same search quality with stable Linux container and GPU-accelerated inference.
ClaireGPT Use Case — RAG Chatbot Evolution
pivot: from search snippets to conversational knowledge agent

CODEC returned page snippets — users still had to read and synthesise across multiple results manually. The next evolution: a full RAG chatbot that retrieves relevant knowledge and synthesises a cited answer in natural language.

The system needed to cover three knowledge domains:

  • Process documentation (SOPs, policies, guidelines)
  • Domain abbreviation definitions (logistics-specific terms)
  • Contract clause data (from the BART project — see BART case study)
Document Ingestion — Docling + EasyOCR + Multi-Source
sources: Confluence · SharePoint · network folder · fallback: PDF + EasyOCR

Built a cron-based ingestion pipeline pulling documents from three sources: Confluence, SharePoint, and a network folder — each with a dedicated processor. Docling converts each document to structured markdown, preserving headings, tables, and layout for cleaner downstream chunking.

For older or unsupported file formats, a fallback pipeline converts them to PDF first, then runs Docling with EasyOCR (EasyOcrOptions(force_full_page_ocr=True)) to handle scanned or image-heavy pages.

Why Docling: Preserves document structure (headings, table layout) as markdown — gives the chunker and embedder richer context than plain text dumps from raw PDF extraction.
Chunking and Embedding Experimentation
5 strategies evaluated · agentic chosen · SFT-Mistral vs SentenceBERT (RAGAS) · indexed via Logstash to ELK

Experimented with all 5 chunking strategies. Documents vary widely — structured SOPs, slides, scanned PDFs, and unstructured text — so no single fixed strategy fits all:

StrategyBehaviourTrade-off
Fixed-sizeSplits by token/character countFast but cuts mid-sentence
SentenceSplits on sentence boundariesBetter coherence, ignores topic shifts
SemanticSplits on embedding similarity dropsTopic-aware, computationally expensive
RecursiveHierarchical split with fallback delimitersReasonable default, misses semantic shifts
AgenticLLM identifies topic boundaries, assigns chunk title + summaryBest retrieval quality; slower; some boundary information loss

Agentic chunking chosen (AgenticChunker) — the LLM generates a chunk title and summary per chunk, improving both retrieval precision and answer synthesis. Trade-off: slower ingestion and some information loss at chunk boundaries when the LLM abstracts content.

Embedding comparison via RAGAS: SFT-Mistral (fine-tuned Mistral embedder) vs SentenceBERT. SFT-Mistral yielded better RAGAS scores on the domain corpus and was selected for production. Each chunk is pushed to Elasticsearch via Logstash with three fields: vector (dense embedding), summary, title.

Key finding: Agentic chunking improved retrieval relevance most significantly. The LLM-generated chunk title gives the retriever richer signal than raw chunk text alone.
Agentic RAG — Single Agent with 3 Tools
framework: LangChain tool-calling agent · 3 tools: abbreviation · knowledge search · BART contract clauses

Built a single LangChain tool-calling agent with three specialised tools, each connecting to a different backend system:

ToolFunctionBackend
fetch_abbrev_explanationAbbreviation finderDomain abbreviation API (NTLM auth) — returns markdown table of matches
fetch_clm_document_chunksKnowledge searchElasticsearch kNN search on vector field — returns top-N chunk title + text + URL
fetch_customer_clausesContract clause searchLLM generates Oracle SQL from query — executes against BART contract clause database

The agent receives the user query and conversation history, selects and calls tools (single or multi-turn), then synthesises a cited response. Role-based access control (TOOL_PERMISSIONS) gates which tools each user role may invoke.

BART integration: fetch_customer_clauses uses LLM-to-SQL generation: the agent passes the natural language query to an LLM which outputs an Oracle SQL statement, executed against the BART-extracted clause database — connecting two independent systems through the agent layer.
Production Deployment — Nginx · React.js · Chat History
frontend: React.js · reverse proxy: Nginx · OpenShift · per-user chat cache + QnA collection

Deployed with a React.js frontend served via Nginx reverse proxy, FastAPI backend in a separate pod on OpenShift. Two usage patterns:

  • Client loading (per-user chat history cache): each user's conversation history is cached client-side and sent with each request — giving the agent full multi-turn context without server-side session management
  • Server loading (QnA collection): every query–response pair with user feedback (like/dislike via FeedbackRequest) is stored server-side in chat_history/ for future fine-tuning and evaluation
Outcome: Production-deployed knowledge chatbot with multi-turn dialogue, role-gated tool access, and structured QnA data collection for continuous improvement.
